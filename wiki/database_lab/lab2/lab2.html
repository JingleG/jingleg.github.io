Filter and Join

按照 instruction 给的顺序看

先是 Predicate

Predicate 里有三个固有参数，分别时 field, op, operand，对应 tuple 中的 列号，操作符 和 要比较的值。更具体的说就是分别对应 table.columnNo = 10。比较的时候是直接调用 field 给的 compare 方法即可。

同理还有 JoinPredicate，只不过是换成两个 tuple 比较， 即 table1.column1 = table2.column2。

然后是 Filter，可以理解为作用在 iterator 上的 filter，即在原本 iterator 的基础上仅返回满足 Predicate 的 Tuple。注意 Operator 类中有 open 状态，需要 super.open()。

最后是 Join，相比前面要稍微麻烦一点，需要通过迭代左右两个 iterator 将两个表 join 起来。实现使用了注释提示的 nested loops join，使用额外的变量记录 当前左表的 tuple，循坏右表进行 join。

同时勘误，lab1中 SqeScan 在 close() 的时候不能将 iterator 置为 null。

Aggregates

实现聚合操作，包含 Count, Sum, Avg, Min, Max，根据不同的操作符统计结果，最后返回 Map 中的结果，即 <aggregateKey, aggregateValue> 的 Tuple 迭代器，具体形式是 OpIterator。
这里思路也很简单，直接每次把值 merge 到 HashMap 即可，但是实现有很多选择。根据 Aggregator 中的注释，暗示使用 TupleIterator 来迭代，但是需要传入 Iterable<Tuple> tuples，也就意味着要额外存一个结果的 List<Tuple>。这里当时考虑有3个办法。

  • Plan A：手写一个类实现 OpIterator，这样可以直接在 map 中迭代实时生成新的 tuple 返回，优雅且不需要额外的空间，缺点是要写很多，而且后面测试如果强制要求使用 TupleIterator 就傻了。
  • Plan B：将 map 中直接使用 Tuple 作为 Value，每次操作的时候改成创建一个新的 IntField 然后再对 Tuple setField()。这样可以直接传 map.values() 给 TupleIterator 而不用再次循环 Map。这个办法感觉省了空间同时也免去了输出结果的额外循环，但是仔细想想也没差别，每次 merge 值的时候常数不知道要翻多少倍。
  • Plan C：直接在 map 里记录结果，返回 iterator 的时候直接在 map 里把值都掏出来放到一个 list 里，丢给 TupleIterator 即可，最省事。

在一通思考之后果断选择 Plan C，不愧是我。
实现相比别的部分也更为复杂，因为有不同聚合类型需要疯狂 if-else，导致巨量冗余代码。写了一半改成 抽出 interface 附加默认方法，到最终 使用抽象类单独实现 merge，总算看上去稍微优雅一点。

对于 Min/Max 这种单一操作，直接 map.merge(key, 1, Integer::sum) 放到map中即可。 唯一不同的是 AVG，使用额外的 sum 和 cnt 记录总和以及个数，同时更新 map 中的结果。

对于 StringAggregator，与 IntegerAggregator 相同，只是仅限 Count 操作。

最后是 Aggregate，可以理解为对 StringAggregator 和 IntAggregator 的封装，传入原始的 tuple 列表，输出聚合的结果迭代器。注意这里需要自己 call mergeTupleIntoGroup() 生成结果，我这里决定放在 open() 到时候生成，副作用是 rewind()，不能再 close(),open()，否则会导致 sum 重新计算。其他可以考虑的解决的方案比如清空 map 重新计算。

HeapFile Mutability

这部分花了不少时间，其中一半是 instruction 说的有点模糊，另一个就是极度怀疑这个 test file 写的有问题。

Lab2 实验代码勘误

lab2 的 BufferPollWriteTest 中最后一个测试 handleManyDirtyPages 使用的 static class HeapFileDuplicates extends HeapFile 的 insert() 方法没有调用 BufferPool.getPage()，而是直接在 new 的 page 做的操作导致 BufferPool 中没有页面过不了测试。

来自 lab 的 instruction

Note that it is important that the HeapFile.insertTuple() and HeapFile.deleteTuple() methods access pages using the BufferPool.getPage() method; otherwise, your implementation of transactions in the next lab will not work properly.

虽然也可以在 Bufferpool 中 insertTuple()/deleteTuple() 拿到修改的 pages 然后手动在放到 pool 中，但让我觉得这不符合设计初衷。原因很简单，每次在 BufferPool 上调用修改的时候，实际是要去调用 DbFile 去执行实际的修改操作，而根据 instruction，在写 page 的时候会反过来用 BufferPool.getPage() 来读取 Page，也就意味着所有的脏页已经都在 BufferPool 中了，不需要再次手动放到 BufferPool 中。

当然，我能想到其他的原因是 tuple 修改了多个页，导致 BufferPool 触发 Page Eviction，让原本在 BufferPool 中的页被刷盘了。但这个也说不过去，因为脏页在完成事务之前应该是不能直接刷盘的，至少我认为在这个简陋的 lab 里应该不行。如果后面有其他的部分过不了没准就再修改。

后面我也随便看了几个别人的解决方案，基本都是 Plan B 这样拿到修改的页之后再标记脏页并放到 Buffer 中。

🤡 Update: 只要 pool 满了就可以直接 evict，脏页写 file 即可，否则直接丢弃。那么 Plan B 这种在修改时不标记 dirty，最后统一标记并放入 pool 中的实现也是可以理解的：在修改的时候，仅修改 page 上 tuple 数据，出现 pool 满了的情况时，由于实际已经修改的页面并没有被标记 dirty，也就不会被 flush 而被直接丢弃。最后返回的 page 集合包含了所有的脏页，统一标记，放入 pool 中，放不下就 evict，但是由于这次已经被标记 dirty 会被写入 file 中，保证 file 和 pool 的数据一致性。总结，可行但效率低下，同一个 page 会被 fetch 到 pool 中两次。

我自己的实现是直接在 DbFile 中修改时就直接标记脏页了。

决定先修改 test file，后面如果出现 BufferPool 满了或者别的问题，再修改成 Plan B。

目标：实现通过 BufferPool 对 HeapFile 以及 HeapPage 上 tuple 的删除与插入操作。
直接看图，大致的流程：

  1. 当要对某个 tuple 进行修改时，会调用 DbFile 进行操作。
  2. DbFile 需要从 BufferPool 获取 page，找到合适的位置然后使用 Page 的抽象进行删除或插入操作，最后将被修改的 pages 返回。
  3. 一个特殊情况时插入时所有的 page 都已经满了，需要手动创建一个新的 page，在新的 page 上插入。这里 lab 的 instruction 给了一个很迷惑的提示。

Implement the remaining skeleton methods in:

  • src/java/simpledb/storage/HeapPage.java
  • src/java/simpledb/storage/HeapFile.java (Note that you do not necessarily need to implement writePage() at this point).

我当时看了很久都不明白为什么可以不实现 HeapFile 的 writePage()。因为如果需要创建一个新的 Page，那么创建完成之后，按照 instruction 的说法，需要调用 BufferPool.getPage() 来获取页面，也就意味着 BufferPool 会调用 ReadPage() 来读 DbFile，不先写岂不是读了个寂寞。我自己实现的时候还是一起实现了 writePage()

其他的基本就是：

  • 删除：从 tuple 中拿到 recordId，找到对应的 Page， 标记 Page Header 中的 bitmap 完成删除。根据注释中的 This tuple should
    be updated to reflect that it is no longer stored on any page     推测大概率需要将 RecordId 置为 null。
  • 插入：从 File 中找到一个有空 slot 的 Page，如果没有空的 Page 则创建新并插入，同时更新 tuple 的 recordId。

其中可能需要斟酌的是在哪里将 page 标记为 dirty，因为代码注释中是在 BufferPool.insertTuple() 上写了要 markDirty bit。但是我的实现是在 DbFile 从 BufferPool 中拿到 Page 并修改之后立刻标记，因为其实当前 buffer 中的页其实已经是脏页了，后面如果出现 page eviction 那么被修改的页要么不能被弹出，要么先写到 disk 再弹出，细节可以看勘误的部分。

Insertion and Deletion

实现 Operator Delete 和 Insert。这部分也没什么难度，直接调用 BufferPool 的方法即可。注意返回的 TupleDesc 是一个单独的 Field 记录 操作影响的 row number。另外 Delete 的 hasNext() 没有和 Insert 一样提示 @return A 1-field tuple containing the number of inserted records, or null if called more than once. 但如果多次执行也要返回 null。

Page Eviction

实现一个 Page Eviction Policy，选择很多

  • FIFO，先进先出
  • LRU，最久未使用
  • LFU，最不常用（使用次数）
  • Clock，时钟置换，在环形链表上轮流标记不活跃的页，轮询到被标记的则清除

题不能白刷： LRU

实现：基本就是直接用 LRU 的封装替换了原有的 map，注意 lab 给的注释即可。

总结

总的来说这部分页不算困难，主要是有些细节 instruction 没有细说，需要看代码猜下一步要怎么做，尤其是 HeapFile，HeapPage 和 BufferPool 的互相调用。其他的部分只要认真看 instruction 就能拿下。
